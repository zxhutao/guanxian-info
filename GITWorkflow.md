## 2. 分支策略

### 📌 采用：主干开发 + 功能分支

```
main ─────●────●────●────●────●─── (始终保持可发布)
           \  /      \  /
            ●        ●            (短期功能分支)
```

### 分支命名规则

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能分支 | `feat/功能名` | `feat/用户登录` |
| 修复分支 | `fix/问题描述` | `fix/页面闪烁` |
| 紧急修复 | `hotfix/问题描述` | `hotfix/登录崩溃` |

## 3. 提交规范

### ✅ 正确示例

```
feat: 添加用户登录功能
fix: 修复招聘列表加载慢的问题
docs: 更新README说明
chore: 升级uni-app版本
style: 调整首页样式间距
refactor: 优化代码结构
```

### ❌ 错误示例

```
update
修改
fix bug
asdf
```

### 提交类型说明

| 类型 | 含义 | 使用场景 |
|------|------|----------|
| `feat` | 新功能 | 添加新页面、新组件 |
| `fix` | 修复 | Bug修复 |
| `docs` | 文档 | 文档更新 |
| `style` | 样式 | 样式调整 |
| `refactor` | 重构 | 代码优化 |
| `chore` | 杂项 | 构建、依赖更新 |

## 4. 工作流程

### 标准流程（人类审核）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  OPC 开始   │ -> │  OPC 开发   │ -> │  OPC 提交   │ -> │  等待审核   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                            |
                                                            v
                     ┌─────────────┐    ┌─────────────┐    │
                     │   你审核    │ <- │  你合并到   │    │
                     │   通过 ✅   │    │   main     │    │
                     └─────────────┘    └─────────────┘    │
                           |                               │
                           v                               │
                     ┌─────────────┐                       │
                     │  发布测试   │ ----------------------
                     └─────────────┘
```

### 详细步骤

#### 第1步：开始新任务
```bash
# 确保代码最新
git pull origin main

# 创建功能分支
git checkout -b feat/功能名
```

#### 第2步：开发与提交
```bash
# 开发完成后，提交代码
git add .
git commit -m "feat: 添加xxx功能"
```

#### 第3步：推送分支
```bash
# 推送到远程
git push origin feat/功能名
```

#### 第4步：人类审核
- 你检查代码
- 通过后合并到main

#### 第5步：清理分支
```bash
# 合并后删除远程分支
git push origin --delete feat/功能名

# 切换回main并更新
git checkout main
git pull origin main
```

## 5. OPC操作指引

> 下面的步骤是给OPC（数字员工）执行的，请严格按照执行！

### ⚡ 快速开始

```bash
# 每天工作前执行（确保代码最新）
git checkout main
git pull origin main
```

### 🔧 开发新功能

```
1. 创建分支: feat/功能名（用中文描述，如：feat/用户登录）
2. 开发代码
3. 提交: feat: 描述你做了什么
4. 推送: git push origin feat/功能名
5. 告诉我完成了，等待审核
```

### 🐛 修复Bug

```
1. 创建分支: fix/问题描述（用中文描述，如：fix/登录页面闪烁）
2. 修复问题
3. 提交: fix: 描述修复了什么
4. 推送: git push origin fix/问题描述
5. 告诉我完成了，等待审核
```

### ⚠️ 重要规则

1. **永远不要直接推送到main分支**
2. **每个提交只做一件事**
3. **提交信息必须包含类型前缀**
4. **完成工作后必须通知我审核**
5. **如果不确定能不能做，先问我**

## 6. 回滚与恢复

### 撤销未提交的修改

```bash
# 撤销所有未提交的修改
git checkout .
# 或者
git restore .
```

### 撤销已提交但未推送

```bash
# 撤销最后一次提交（保留修改）
git reset --soft HEAD~1
```

### 查看历史

```bash
# 查看提交历史
git log --oneline -10

# 图形化查看
git log --graph --oneline --all
```
